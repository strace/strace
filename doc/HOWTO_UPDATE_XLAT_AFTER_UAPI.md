# How to Update xlat Files After a Linux UAPI Change

This document describes a practical workflow for refreshing strace’s
`src/xlat/*.in` translation tables when the Linux kernel’s userspace API
headers change (new `ioctl` values, `netlink` attributes, `CLONE_` flags,
`#define` constants, and so on).  It is written for human maintainers and
for agents that need a repeatable, checkable process.

## Context and related documentation

- **Xlat file format, directives, and generation**
  — [doc/README-xlat.md](README-xlat.md) and
  [src/xlat/gen.sh](../src/xlat/gen.sh).
- **Commit message style (including `xlat:` updates)**
  — [doc/COMMIT-MESSAGES.md](COMMIT-MESSAGES.md) (see “Xlat Constant
  Updates” and `xlat: update …` examples).
- **Regenerating enum-driven xlats**
  — [maint/update-xlat.sh](../maint/update-xlat.sh) (for files with
  `#Generated by maint/enum2xlat.sh` or similar — not covered in depth
  here; run only when the corresponding generator line is present).

## Why maintain `#From` / `#Prefix` / `#Pattern`?

The maintenance scripts
[maint/list-xlat-linux-headers.sh](../maint/list-xlat-linux-headers.sh) and
[maint/find-new-xlat-constants.sh](../maint/find-new-xlat-constants.sh)
**only know about headers that are tied to an xlat file** via:

1. A `#From <path>` line (path relative to the **Linux** tree given by
   `-d`, e.g. `include/uapi/linux/sched.h`); and/or
2. A line `#Generated by maint/gen_xlat_defs.sh …` or
   `#Generated by maint/enum2xlat.sh …` (as produced by the generators).

**Optional** metadata used by the finder (ignored by
[src/xlat/gen.sh](../src/xlat/gen.sh), but read by
`find-new-xlat-constants.sh`):

- `#Prefix P1_ P2_` — only names matching one of the prefixes (regex
  composition) are compared.
- `#Pattern R1 R2` — like prefix, but full EREs (see script usage text).

If an xlat lists constants that are not described with `#From` or a
`#Generated` line, the scripts will not associate them with a header and
**will not suggest** new values when that header changes.  Converting
informal comments to `#From` (and, where needed, `#Prefix` / `#Pattern`)
makes later UAPI sweeps much easier.  A prior example in the project:
“xlat: convert comments about source of xlat constants into directives”
(see [doc/COMMIT-MESSAGES.md](COMMIT-MESSAGES.md)).

## Commit range syntax (Git)

Both scripts that take a **commit range** require **two-dot** form:

- **Use:** `v7.0..v7.1-rc1` or `abc123..def456`
- **Do not use:** `v7.0...v7.1-rc1` (three-dot) — the scripts exit with
  an error.
- A **single** revision is not accepted; use `COMMIT^..COMMIT` or
  `BASE..BASE` to express one commit if needed.

`list-xlat-linux-headers.sh` runs `git diff` in the **Linux** repository
from `COMMIT1` to `COMMIT2` to see which UAPI (or other) **header files**
changed, then **filters** the table to rows whose header is in that diff.
So `v7.0..v7.1-rc1` means “all commits reachable from v7.1-rc1 that are
**not** in v7.0’s history” (same as `git log v7.0..v7.1-rc1`).

## Step 1 — List kernel headers tied to your xlats (optional filter)

**Purpose:** build a TSV mapping each relevant `*.in` file to the
Linux **header** path(s) you care about, optionally restricted to headers
that **changed** in a given Git range.

**Prerequisites:** a local clone of the Linux tree (`LINUX` below).

From the strace top-level directory:

```sh
maint/list-xlat-linux-headers.sh -d "$LINUX" -t -c v7.0..v7.1-rc1 \
  src/xlat/*.in > xlat-linux-headers.tab
```

Options:

- `-d LINUX` — path to the Linux **kernel source** (must contain
  `include/uapi/`).  **Mandatory.**
- `-t` — table output: `xlat_file`, `line_type` (`gxd` / `enum` /
  `from`), `header_path` (tab-separated).
- `-c COMMIT1..COMMIT2` — only keep rows whose `header_path` changed in
  that **two-dot** range.  **Requires** `-t`.

**What `line_type` means (for the next step):** carried through for
consistency; `find-new-xlat-constants.sh` currently only needs columns 1
and 3 (`xlat_file` and `header_path`).

**Observed on one maintainer run (v7.0 to v7.1-rc1, full `src/xlat/*.in`):
~175 table rows; `list-xlat-linux-headers.sh` took on the order of
10s CPU I/O on a typical dev machine (exact time depends on disk and tree
size).**

If you omit `-c`, you get the full set of header paths (no Git diff
filter) — useful when bootstrapping `#From` lines or when comparing
across a non-Git drop of headers.

## Step 2 — Propose new constants to add to each xlat

**Purpose:** for each row in the Step 1 table, compare the header at the
**old** and **new** ends of the range, list all `#define` / `enum` names
that the script considers in scope, and print names that are **in the new
header**, **not** in the old, and **not** already listed in the
corresponding strace `*.in` file.

```sh
maint/find-new-xlat-constants.sh -d "$LINUX" v7.0..v7.1-rc1 \
  < xlat-linux-headers.tab > new-xlat-constants.tab
```

- Reads the **table from stdin** (as produced in Step 1 with `-t`).
- The commit range is a **single positional** argument: `v7.0..v7.1-rc1`
  (same two-dot rules as above).

**Heuristic in brief:** the script looks at the existing **constant
names** in the xlat (stripping `1<<` for shift entries).  It infers
prefixes either from a `#Prefix` line, the **longest common prefix** of
the names, or a heuristic from the file name.  Alternatively it uses
`#Pattern`.  It then:

1. Extracts all matching identifiers from the header as of `COMMIT1` and
   `COMMIT2` (via `git show` — `#define` and a subset of `enum` lines),
2. Forms `new = in(v2) \ in(v1) \ xlat_set`,
3. Emits: `xlat.in<TAB>header.h<TAB>name1 name2 …`

**Observed (same v7.0..v7.1-rc1, full xlat set): ~24 output lines, ~8s
runtime for the finder** on the same kind of host as above.

**Important limitations (false positives and blind spots):**

- **One header, many UAPI “namespaces”.**  The extractor walks the whole
  file.  If `if_link.h` (for example) contains the **top-level** `IFLA_*`
  link enum **and** nested `IFLA_BR_*`, `IFLA_BOND_SLAVE_*`,
  `IFLA_NETKIT_*`, the script may report `IFLA_BOND_SLAVE_*` for an xlat
  that only models the **device**-level `IFLA_*` list.  The maintainer
  must map each name to the **right** strace xlat and nested decoder.
- **Constants that are not plain `#define` / simple `enum` members** can
  be missed or misread.
- **Same new name in two xlats:** e.g. a mask and a key type that share
  a `BTRFS_` name — the script can flag both; one may be semantically
  wrong for that xlat.
- **Prefix too broad for `K_` (keyboard) xlats:** a new `K_CSI_*` family
  may appear in every `kd_key_*.in` that uses `#Prefix K_`.  Usually only
  **one** file (or a new dedicated `*.in`) should list those keys; the
  rest are noise.  Tighten with `#Prefix` / `#Pattern` on the xlat, or
  add the keys once and ignore duplicate rows in the table.
- **`#Pattern` / `SCHED_` and policies vs flags:** the script may suggest
  `SCHED_GETATTR_FLAG_*` for [src/xlat/schedulers.in](../src/xlat/schedulers.in)
  if that file uses a broad `SCHED_` pattern while it actually **indexes
  scheduling policy IDs**, not `sched_getattr` flag bits.  The constant
  may still belong in a **new** or different xlat when the decoder is
  updated to print symbolic flags.

## Step 3 — Manual triage and edits

1. **Open** `new-xlat-constants.tab` one line at a time.
2. For each name, check the **kernel header** in context (and the strace
   decoder, if any): is this constant part of the **same** UAPI
   “bucket” that this xlat models?
3. **Add** names to the appropriate `src/xlat/*.in` following
   [doc/README-xlat.md](README-xlat.md) (ordering: `#value_indexed` and
   `#sorted` have extra constraints; **never** break sort / index
   invariants without checking `gen.sh` and tests).
4. If the constant is **genuinely new in scope** but no xlat or decoder
   yet exists, the change may need a new `*.in`, an include in
   `src/defs.h` if the table is shared, decoder updates, and tests — same
   spirit as [doc/HOWTO_ADD_SYSCALL.md](HOWTO_ADD_SYSCALL.md).

### Bundled headers and buildability

strace often ships a **sanitized** copy under `bundled/linux/`; many
xlats are written so that if a symbol is missing from the **build**’s
headers, a **numeric fallback** on the same line still makes the
generated [src/xlat/*.h](../src/xlat/) self-consistent.  If you add only a
**name** and no fallback, and the build environment’s headers (or
bundled) are older than the kernel you diffed, adjust **bundled UAPI** or
use the `NAME  value` fallback form; see
[doc/COMMIT-MESSAGES.md](COMMIT-MESSAGES.md) under “Bundle Header Updates”.

### When `maint/update-xlat.sh` applies

Use [maint/update-xlat.sh](../maint/update-xlat.sh) only for files with
`#Generated by maint/enum2xlat.sh …` at the top — that script re-runs the
enum generator.  It does not replace the Step 1–2 workflow for hand-edited
xlats.

## Step 4 — Regenerate generated xlat headers and test

- After changing `*.in`, a normal `make` (or your usual build) should
  regenerate the corresponding `*.h` if your tree is set up with the usual
  rules.  If you add **new** `*.in` or change build wiring, you may need
  `./bootstrap` and autotools regeneration as in
  [doc/HOWTO_ADD_SYSCALL.md](HOWTO_ADD_SYSCALL.md).
- **Tests:** if a test compares strace’s symbolic output to expected
  strings, update it when a constant becomes decodable.  A practical
  signal: use `git log` on a given `src/xlat/foo.in` — if past commits
  for that file touched `tests/…`, the next UAPI change likely will too.
- Run a full check to identify all failing tests, e.g.
  `./bootstrap && ./configure && make -s -j$(nproc) && make -s -j$(nproc) check VERBOSE=1`.
- After fixing tests, run a focused check of those tests that were failing
  before the fix, e.g.
  `make -s -j$(nproc) check VERBOSE=1 TESTS='clone3.gen.test ioctl_kd.gen.test'`.

### Tests beyond the obvious `.gen` matches

Some xlats affect tests that **do not** fail until you grep for related
symbols:

- **Bitmask / composite literals:** Search `tests/` for OR-ed masks that
  include the **previous** full set of bits for that subsystem (e.g.
  `LANDLOCK_ACCESS_FS_*` combined in [tests/landlock_add_rule.c](../tests/landlock_add_rule.c)
  and [tests/landlock_create_ruleset.c](../tests/landlock_create_ruleset.c)).
  Add any new access-bit constant to those literals when you add it to
  [src/xlat/landlock_ruleset_access_fs.in](../src/xlat/landlock_ruleset_access_fs.in).

### Pitfalls seen in review (avoid next time)

These mistakes are easy to miss when only regenerating expectations from a
first failing `make check` run:

1. **`tests/clone3.c` magic masks vs `clone3_flags.in`.**
   The clone3 test uses **fixed** `uint64_t` masks chosen years ago so that
   `printflags_ex` exercises particular decode paths.  After adding new
   `CLONE_*` entries to `clone3_flags.in`, you often must **adjust the hex
   literals** in `tests/clone3.c` so that:
   - every clone3-only flag you intend to appear in the golden string is
     actually **set** in the mask (including high-bit flags such as
     `CLONE_PIDFD_AUTOKILL`); and
   - the “unknown remainder” case still prints `CLONE_???` for leftover
     bits — avoid replacing `XLAT_UNKNOWN(...)` with `XLAT_KNOWN(...)`
     unless you deliberately rebuild the mask for that scenario.
   Compare with actual `strace` output from the failing test run when in doubt.

2. **`tests/fsmount.c` flag rows vs sparse enums.**
   Do not merge unrelated numeric rows into a single symbolic decode string.
   When the kernel adds one flag (e.g. `FSMOUNT_NAMESPACE`) but leaves gaps,
   keep **separate** table rows for other probe values (e.g. `4`,
   `0xfffffffc`) as `FSMOUNT_???` until those bits have names in UAPI.

3. **`tests/ioctl_kd.c` and `K(KT_*, …)` after `kd_key_types.in` changes.**
   Keyboard values use `K(type, value)` (high byte = key type).  After adding
   `KT_CSI`, update `kb_value_vecs[]` examples so **KT_CSI** lines use the
   correct encoding (e.g. `K(KT_CSI, 0xc)` not ambiguous legacy `K(0xf, …)`
   forms), and retain coverage for other key-type lanes if you replace rows.

4. **`tests/netlink_nlctrl.c` and genl command enums.**
   The `nl80211_cmds[]` array ends with `{ LAST_CMD + 1, "NL80211_CMD_???" }`.
   When you append commands to [src/xlat/genl_nl80211_cmd.in](../src/xlat/genl_nl80211_cmd.in),
   set **`LAST_CMD` to the last named enum member you added**, not an older
   sentinel — otherwise the fake “unknown” command ID drifts.

5. **`NEWS` wording.**
   Follow the style of surrounding bullets: short **family** names
   (`KVM_*`, `NETDEV_*`, `NL80211_*`) rather than only the longest exact
   prefixes (`KVM_CAP_*`, `NETDEV_CMD_*`, `NL80211_CMD_*`) unless the release
   notes already use the long form consistently nearby.

## Step 5 — Commit message, `NEWS`, and commit split

- Follow [doc/COMMIT-MESSAGES.md](COMMIT-MESSAGES.md):
  - Title: e.g. `xlat: update FOO_* constants` or
    `xlat: update FOO and BAR table entries`.
  - **ChangeLog-style** bullets per file, with
    `(NEW_SYMBOL):` and “New constant introduced by Linux kernel commit
    …” when the kernel point is clear (`git describe --match 'v*'
    --contains` on the **Linux** side).
- Mention user-visible improvements in [NEWS](../NEWS) when the change
  is user-visible.  Commit those `NEWS` edits **in the same commit series**
  as the related `src/xlat/*.in` and test changes (same commit as each
  logical chunk, or one summary `NEWS` commit at the end that matches the
  tables you touched)—not as a detached documentation-only change.
- Group updates by the **basename of kernel headers** referenced in
  `new-xlat-constants.tab` (column 2), then by related
  **prefixes/suffixes/patterns** from that table (for example, families
  like `IFLA_*` vs `IFLA_BR_*`, or similarly named but different UAPI
  buckets in one header).
- Use this grouping as the default commit split strategy for xlat refreshes,
  instead of broad subsystem buckets.  This usually yields reviewable commits
  with a clear “one header family / one naming pattern” scope.
- Before finalizing, check recent history of `src/xlat/*.in` updates for
  concrete precedents and naming style, e.g. with `git log -- src/xlat/*.in`.

## Example: v7.0 to v7.1-rc1 (pilot)

Commands (from strace top-level, `LINUX` pointing at a full Linux tree
with tags `v7.0` and `v7.1-rc1` — **verify** with
`git -C "$LINUX" rev-parse v7.0 v7.1-rc1`):

```sh
maint/list-xlat-linux-headers.sh -d "$LINUX" -t -c v7.0..v7.1-rc1 \
  src/xlat/*.in > xlat-linux-headers.tab
maint/find-new-xlat-constants.sh -d "$LINUX" v7.0..v7.1-rc1 \
  < xlat-linux-headers.tab > new-xlat-constants.tab
```

**Approximate size (one run, full `src/xlat/*.in`):**

| Stage                         | Count / time (indicative)   |
|------------------------------+-----------------------------|
| `xlat-linux-headers.tab` rows| ~175                        |
| `new-xlat-constants.tab` rows| ~24                         |

**Representative false positives in that `new-xlat-constants` output
worth ignoring or handling specially:**

- **`rtnl_link_attrs.in` + `include/uapi/linux/if_link.h`:** new
  `IFLA_BOND_SLAVE_*` / `IFLA_NETKIT_*` / `IFLA_BR_STP_MODE` (when the
  strace `IFLA_` xlat is only the **link-level** `enum` from the start of
  the file) — the **link-level** `IFLA_` set did not gain entries in
  v7.0..v7.1-rc1; the interesting additions live under the **bridge** /
  **bond** / **netkit**–specific xlats and their decoders in
  [src/rtnl_link.c](../src/rtnl_link.c) and friends.
- **`BTRFS_FREE_SPACE_FLAGS_MASK` for two different btrfs xlats** —
  often not the same semantic class as the surrounding key/objectid
  table.
- **Keyboard `K_CSI_*` on many `kd_key_*.in` files** — one logical place
  to add the keycap lines (or a new `*.in`); the rest of the table rows
  are duplicates.
- **`SCHED_GETATTR_FLAG_DL_DYNAMIC` for `schedulers.in`** — policy
  xlat, not `sched_getattr` flags; needs a different table or `sched.c`
  if you want that symbol in decoded output.
- **PID `PIDFD_INFO_*` vs [src/xlat/pidfd_open_flags.in](../src/xlat/pidfd_open_flags.in)** — open
  flags vs get-info **mask** bits; only **open**-style symbols belong in
  `pidfd_open_flags.in`.

A **concrete, merged** subset of updates that are straightforward from
this range (as applied while writing this how-to) include:

- [src/xlat/clone3_flags.in](../src/xlat/clone3_flags.in) — `CLONE_AUTOREAP`,
  `CLONE_NNP`, `CLONE_PIDFD_AUTOKILL`, `CLONE_EMPTY_MNTNS` (clone3/64-bit
  flag space; see
  [src/clone.c](../src/clone.c) `printflags_ex(..., clone_flags,
  clone3_flags, ...)`).
- [src/xlat/fsmount_flags.in](../src/xlat/fsmount_flags.in) —
  `FSMOUNT_NAMESPACE`
- [src/xlat/landlock_ruleset_access_fs.in](../src/xlat/landlock_ruleset_access_fs.in) —
  `LANDLOCK_ACCESS_FS_RESOLVE_UNIX` (re-run landlock tests if the mask
  string expectations change)
- [src/xlat/genl_netdev_cmd.in](../src/xlat/genl_netdev_cmd.in) —
  `NETDEV_CMD_QUEUE_CREATE` (after `NETDEV_CMD_BIND_TX`, value
  21 in the uapi `enum` starting at 1)
- [src/xlat/kvm_cap.in](../src/xlat/kvm_cap.in) —
  `KVM_CAP_S390_VSIE_ESAMODE` `248` (number column matches kernel
  [include/uapi/linux/kvm.h](https://github.com/torvalds/linux/blob/master/include/uapi/linux/kvm.h))
- [src/xlat/pidfd_open_flags.in](../src/xlat/pidfd_open_flags.in) —
  `PIDFD_AUTOKILL` with fallback `O_TRUNC` (as in the kernel uapi
  [pidfd.h](https://github.com/torvalds/linux/blob/master/include/uapi/linux/pidfd.h))
- [src/xlat/kd_key_types.in](../src/xlat/kd_key_types.in) — `KT_CSI` with
  value 15 in the `KT_*` value_indexed list

Larger or riskier follow-ups for the same kernel delta include: filling in
[NL80211_CMD_*](../src/xlat/genl_nl80211_cmd.in) additions (tight
`#value_indexed` ordering across a very large enum), bridge
`IFLA_BR_FDB_*` / `IFLA_BR_STP_MODE` next to
[rtnl_ifla_info_data_bridge_attrs.in](../src/xlat/rtnl_ifla_info_data_bridge_attrs.in) **and** the
parallel decode table in
[src/rtnl_link.c](../src/rtnl_link.c), netkit/bond **nested** attribute
xlats, and a dedicated place for the `K_CSI_*` keycap `K(KT_CSI, …)` lines.

## Quick checklist (for agents)

1. [ ] `LINUX` clone has both endpoint tags/commits; use **two-dot** range.
2. [ ] Run `list-xlat-linux-headers.sh -d … -t -c OLD..NEW src/xlat/*.in`.
3. [ ] Run `find-new-xlat-constants.sh -d … OLD..NEW` with that TSV on stdin.
4. [ ] Triage: drop nested-enum / wrong-xlat / wrong-prefix lines; adjust
   `#From` / `#Prefix` / `#Pattern` if the same noise repeats every release.
5. [ ] Edit `src/xlat/*.in`; respect `#sorted` / `#value_indexed` / `#unconditional` rules in README-xlat.
6. [ ] Build; fix bundled headers or fallbacks if the tree does not
   define a new symbol.
7. [ ] Update tests and `NEWS` when output changes; commit with
   `xlat: …` message per [doc/COMMIT-MESSAGES.md](COMMIT-MESSAGES.md).
8. [ ] Re-scan tests: `grep` for composite masks (Landlock), crafted
   clone3 masks, netlink “last command + 1”, and ioctl_kd keyboard macros;
   align with **Pitfalls seen in review** earlier in this document; run `make check` again.
